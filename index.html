The problem

“Tests pass” is not “intent satisfied”

Code generation is solved enough. Intent-conformance is not. A unit test checks an example — it can stay green while the code quietly betrays what you meant.

The mutant is a plausible wrong implementation. The unit test can’t tell it apart from the right one. A validity-aware eval can.

The idea

An eval earns trust: GREEN-on-right ∧ RED-on-wrong

The unit of trust isn’t a passing test — it’s a trustworthy eval. An eval is trusted only when it passes on the intended code and is proven to FAIL on a plausible, intent-violating mutant. That is the structural answer to “who evals the evals?”

The kill record

The mutant the eval catches is its evidence of specificity. A vacuous eval (assert True) is green-on-right — but green-on-wrong too, so it’s rejected. The mutant author works behind an information barrier: it sees the intent and the code, never the eval — so it can’t craft a strawman.

How it works

The repository is the memory

Every fact is PRIMITIVE (authored once) or DERIVED (a pure function of disk). Phase, coverage, eval-trust, and the next action are derived — never stored — so a fresh session always computes the same “what’s next?” and the work survives context rot.

Spec

Numbered intent statements — what you mean, not how. Each binds ≥1 eval.

Trust

Each eval proven GREEN-on-right ∧ RED-on-wrong, behind the eval-blind mutant barrier.

Execute

Implement the slice; the trusted, red eval is the executor’s target.

Verify

Run the suite + a bounded intent-audit. The gate blocks on required evals.

Honesty

What is proven vs. designed

The framework holds itself to its own standard. Here is exactly what runs in the repo today, and what is designed but not yet executed.

Proven · runs in the repo

  • Eval-trust on real code: a mutant green on the unit tests, red on the eval
  • The state deriver is tested (6/6); resumability shown by two cold agents
  • The framework dogfoods itself — its own state resumes from disk

Designed · not yet run here

  • The Tier-2 factory’s unattended run (driver written, not executed)
  • A real mobile device journey; a live different-model judge
  • The worked SaaS & payment-gateway examples are illustrative walkthroughs

A vague intent still yields vague code — the intent-audit surfaces that, it doesn’t fix it for you. See KNOWN-LIMITATIONS.md.

Code generation was never the hard part.

Knowing it built what you meant is — and the eval is what knows.
